# o2 (Fuel L2 spot exchange)

o2 is a spot exchange built on Fuel (a non-EVM L2). The signing format is Fuel-native — **not EIP-712** — and the account model uses delegated session keys created by the owner wallet.

This doc focuses on the integration-time gotchas that distinguish o2 from EVM-style venues. Read alongside the official o2 SDK source (the Python SDK is well-designed and worth reading even if you're implementing in another language).

## Architecture vs an EVM venue

| Aspect             | EVM venue (Vertex-style)                | o2 (Fuel L2)                                  |
| ------------------ | --------------------------------------- | --------------------------------------------- |
| Chain              | Ethereum L2 (EVM)                       | Fuel (non-EVM)                                |
| Signing            | EIP-712 typed data, secp256k1            | Fuel personal message format / raw sha256, secp256k1 |
| Key model          | One trading key (or linked delegated key) | Owner key (creates session) + session key (trades) |
| Order submission   | Sign action, POST to gateway             | Sign action, POST to gateway                  |
| Batch              | One action per request                   | Up to 5 actions per request                   |
| Withdrawal         | Trading key cannot withdraw              | Session key cannot withdraw; **owner key can** |

The key practical difference: o2 has the highest-trust credential of the four-venue set. The **owner key can withdraw funds**, so it should never be loaded into a long-running bot process. The pattern is to use the owner key only at session-creation time, then trade with the session key.

## Owner vs session keys

The two-key system:

- **Owner key**: creates the account, deposits funds, creates sessions. **Can withdraw.** Never used directly for trading.
- **Session key**: ephemeral, scoped to specific markets, created by the owner. Cannot withdraw. Used for all trading.

Session lifecycle:

1. Generate a new secp256k1 keypair (the session key).
2. Owner key signs a session-creation message (in Fuel personal message format — see below).
3. Submit via `PUT /v1/session`.
4. Use the session key for `POST /v1/session/actions` thereafter.

The Python SDK handles all of this; in other languages, replicate the SDK's behavior carefully.

**One session per account**: creating a new session **revokes the previous one**. If you have multiple bot instances, they must share the session, or use separate accounts.

## Signing format — Fuel, not EIP-712

Two distinct signing formats are used.

### Owner signing (session creation, withdrawals, etc.)

```text
message = nonce(u64) + chainId(u64) + funcNameLen(u64) + funcName(bytes) + args(bytes)
hash    = sha256("\x19Fuel Signed Message:\n" + len(message) + message)
signature = secp256k1_sign(owner_private_key, hash)
```

The `"\x19Fuel Signed Message:\n"` prefix is the Fuel equivalent of Ethereum's `"\x19Ethereum Signed Message:\n"`. **Don't substitute the Ethereum prefix.**

### Session signing (trade execution)

```text
hash = sha256(nonce(u64) + numCalls(u64) + callBytes)
signature = secp256k1_sign(session_private_key, hash)
```

**No Fuel prefix.** Raw sha256 over the encoded action body. Mixing the two formats up — using the personal-message prefix for trade actions, or omitting it for owner actions — produces signatures that the venue silently rejects.

## Nonce management — strictly sequential, re-fetch every time

o2 nonces are strictly sequential per account and **never reset**. There is no retry-with-same-nonce semantics.

```text
Always re-fetch the nonce from the contract immediately before signing each action.
```

If a nonce is cached and becomes stale (because another action somewhere incremented it), the transaction is rejected with a generic error. The Python SDK exposes `client.refresh_nonce(session)` for the recovery path.

In other languages, query the contract directly for the current nonce. Build retry logic that **fetches a new nonce on every retry**, not retries-with-the-same-nonce.

## Spot balance constraints — this is the #1 operational issue

On a spot exchange, you can **only sell what you own**. There is no short-selling.

If your strategy sells an asset on o2 as a hedge, you must **pre-purchase inventory**. Selling without holding produces an error at submission time — there is no margin or borrow mechanism on this venue.

Balance check before quoting:

- To place a **buy** order: `quote_currency.unlocked_balance >= price × qty`
- To place a **sell** order: `base_currency.unlocked_balance >= qty`

"Unlocked" means not committed to other open orders. Pull from:

```text
GET /v1/account/{trade_account_id}/balances
→ balances["ETH"].total_unlocked     // available to sell
→ balances["USDC"].total_unlocked    // available to buy with
```

For a cross-venue spreader where o2 is one leg: this constraint dictates which leg can be active. If you have only a USDC-denominated balance on o2, you can only **buy** on o2; the active quoting leg must be set up so that any fill on the opposite venue requires you to **sell** on o2 (impossible without inventory) → you'll lose hedges. Plan inventory at strategy design time, not at runtime.

## Minimum order size

Notional-based:

```text
min_order = 1_000_000_000  // 1.0 USDC in 9-decimal representation
```

So `price × qty >= $1.00`. Much smaller than EVM-venue minimums.

## Decimal scaling — 9 decimals across all assets

o2 uses **9 decimal places** for both base and quote on every asset:

- 1.0 USDC = `1_000_000_000`
- 0.05 ETH = `50_000_000`
- Price of $2148.50 = `2_148_500_000_000` (price × 10^9)

This is uniform across assets, unlike EVM venues where each token's decimals vary. But it **differs from Vertex-style X18 venues**: be very careful in cross-venue code that scales for the wrong base.

The SDK exposes helpers like `market.scale_price()` / `market.format_price()` that hide this; use them rather than scaling by hand.

## Market pair naming

```text
Mainnet: ETH/USDC, wBTC/USDC, FUEL/USDC, USDT/USDC, MOOR/USDC
Testnet: fETH/fUSDC, fwBTC/fUSDC, fFUEL/fUSDC, ...  (testnet assets prefixed with 'f')
```

Use the `get_markets()` response to get the correct `contract_id` per market. Don't hardcode them.

## Order types

```python
OrderType.SPOT          # regular limit (GTC)
OrderType.MARKET        # fills at any price
OrderType.FILL_OR_KILL  # must fill entirely or rejected
OrderType.POST_ONLY     # passive only, rejected if it would cross
OrderType.BOUNDED_MARKET # market within a price tolerance
```

For hedge orders in a cross-venue strategy: **`FILL_OR_KILL`** is usually the right call. Partial fills on hedges create additional state to manage; FOK either gives you a clean hedge or a clean failure (which you retry with more aggressive pricing).

## Batch actions — up to 5 per request

o2 supports atomic batches of up to 5 actions:

```python
batch = (
    client.actions_for("ETH/USDC")
    .settle_balance()
    .cancel_order(old_buy_oid)
    .cancel_order(old_sell_oid)
    .create_order(OrderSide.BUY, buy_price, qty, OrderType.SPOT)
    .create_order(OrderSide.SELL, sell_price, qty, OrderType.SPOT)
    .build()
)
result = await client.batch_actions(session=session, actions=[batch])
```

**`SettleBalance` is required at the start of a batch.** It is not auto-prepended like it is in some single-order paths. Forgetting it produces opaque rejections.

The atomic batch is the right primitive for cancel-then-replace quote refreshes — it avoids a window where you've cancelled but not yet replaced.

## Fill detection — WebSocket is reliable

```python
async for update in client.stream_orders(account.trade_account_id):
    for order in update.orders:
        if order.close:
            # Filled or cancelled — distinguish via order.status or your own tracking
            ...
```

Sub-100 ms typical. This is the right primitive for fill detection. Polling is too slow for cross-venue hedging.

## Session expiry — plan for renewal

Sessions are time-bound. Practical durations:

- Automated bots: 24–48 hours
- High-frequency: 7 days

**Implement proactive renewal** — renew ~5 minutes before expiry. If the session expires mid-trade, your hedge will fail. The renewal itself uses the owner key, so you'll need it accessible (or a process that triggers a flow involving the user).

## Common pitfalls

| Symptom                                            | Likely cause                                                |
| -------------------------------------------------- | ----------------------------------------------------------- |
| Signature rejected on session create               | Used raw sha256 instead of `"\x19Fuel Signed Message:\n"` prefix |
| Signature rejected on trade                        | Used Fuel prefix instead of raw sha256                       |
| Action rejected after a few successful ones        | Stale nonce — must re-fetch every time                       |
| "NotEnoughBalance" on a sell hedge                 | Spot venue; no short-selling; pre-purchase inventory         |
| Batch rejected without a clear reason              | Missing `SettleBalance` at start of batch                   |
| Order placed on wrong market                       | Hardcoded `contract_id` instead of fetching from `get_markets` |
| Old session unexpectedly invalid                   | A new session was created elsewhere — only one per account   |

## Endpoint surface

```text
PUT  /v1/session                  — create session (owner-signed)
POST /v1/session/actions          — execute trades (session-signed batch)
GET  /v1/markets                  — list markets
GET  /v1/depth/{market_id}        — order book snapshot
GET  /v1/account/{id}/balances    — account balances
WS   /v1/ws                       — real-time order/depth updates
```
